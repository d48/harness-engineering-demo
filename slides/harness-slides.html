Why this matters

Recursive self-improvement is an old idea with a new bottleneck

“An ultraintelligent machine could design even better machines… the intelligence of man would be left far behind.”
— I. J. Good, “Speculations Concerning the First Ultraintelligent Machine”, 1965
  • Modern RSI isn't just "model trains a better model" — it includes the model improving its own training pipelines and deployment systems.
  • The same base model performs wildly differently depending on the system wrapped around it.
product ≈ model × harness # same weights, different harness: chat completion → toy Claude Code → ships PRs auto-research → runs experiments

Weng's claim: near-term self-improvement gains come as much from harness engineering as from raw intelligence.

I. J. Good's 1965 "intelligence explosion" framing is the classic RSI story. Weng's update: before models retrain themselves, there's a nearer rung — models improving the scaffolding they run in.

Point at the equation: everyone in the room has felt this — GPT-class weights in a chat box vs the same weights inside a coding agent are different products entirely. That delta IS the harness.

Definition

The harness is everything around the weights

Weng's definition, nearly verbatim: the harness orchestrates execution and decides how the model thinks and plans, calls tools and acts, perceives and manages context, stores artifacts, and evaluates results.

Note the analogy she draws to operating systems: deliberately simple, generic designs that lean on existing software engineering practice — not exotic ML machinery.

Evolution of the idea

From “agent = LLM + tools” to harness engineering

2023 — agent frameworks

agent = LLM + memory + tools + planning + action

Components named, but glue, loops, and failure handling left as an exercise.

2026 — harness engineering adds

  • Workflow design — loop engineering: when to plan, act, test, retry.
  • Evaluation — verifiers wired into the loop, not bolted on after.
  • Permission controls — what the agent may touch, and when to ask.
  • Persistent state management — artifacts and memory that outlive the context window.

The 2023 framing (Weng's own famous "LLM Powered Autonomous Agents" post!) enumerated parts. What changed: the connective tissue became the discipline — loop engineering, evaluation, permissions, durable state.

Analogy for engineers: 2023 was "here are the nouns"; 2026 is "here is the operating system".

Design pattern 1 of 3

Workflow automation — the agent loop

python3 demo/run.py 1 — loop fixes a real failing test, then re-verifies

The goal-oriented cycle: plan → execute → observe/test → improve → execute again, until the goal is met. Same simplified loop as OpenAI's Codex agent diagram and Karpathy's autoresearch repo.

Key line from the article: models analyze trajectories and failure cases through an agent runtime, not a static prompt template. The loop is what makes "observe before you claim success" enforceable.

DEMO 1: sandbox repo with a buggy median(); scripted model, real files, real test run. The harness re-runs the tests itself at the end — never trust the model's claim.

Design pattern 2 of 3

File system as persistent memory

  • Long-horizon work generates artifacts — logs, diffs, summaries, error traces — that outgrow any context window.
  • Context = working memory (fast, small, evicted). Disk = long-term memory (durable, searchable, cheap).
  • File ops are a core LLM skill — so better models manage their own memory better, for free.
python3 demo/run.py 2 — same task twice; only the journaled run survives a context reset

Problem: agentic jobs produce more state than fits in context. Solution: durable file-based state instead of carrying the whole workflow in the window.

DEMO 2 is an A/B: an audit task with a 3-message window. Run A keeps findings in context only — they literally scroll out, and the final answer is wrong. Run B journals to NOTES.md, gets hard-reset mid-task, then a *fresh session* reads the notes and finishes correctly.

This is exactly why Claude Code writes CLAUDE.md, plans, and scratchpads to disk.

Design pattern 3 of 3

Sub-agents and background jobs

python3 demo/run.py 3 — three real threads triangulate a production bug

Use cases from the article: searching multiple hypotheses, concurrent experiments, delegating isolated subtasks — without polluting the main context.

The critical design choice: parallelism is explicit and inspectable through file-based storage rather than transient chat contexts. The parent manages jobs like processes: launch, inspect logs, cancel, merge.

DEMO 3: a crash-looping payment worker. Three sub-agents (real threads) check logs, code, and queue data in parallel; each writes findings/*.md; the orchestrator synthesizes the root cause from the three files.

Part two — harness optimization

The optimization ladder: what we tune keeps moving up

The article's progression: instruction prompts → structured context → workflow → harness code → optimizer code. Each step raises the abstraction: from tuning answers, to tuning the machinery that produces answers, to tuning the machinery that improves the machinery.

Her prediction: harness engineering becomes meta-methodology — fewer heuristics, more general mechanisms — and mature harnesses enable auto-research loops while smarter models prevent over-engineering.

Rung: structured context

ACE — context as an evolving playbook, not a longer prompt

Agentic Context Engineering (Zhang et al., ICLR 2026). Three roles: generator produces trajectories referencing playbook bullets; reflector extracts lessons from successes AND failures; curator merges structured (id, description) updates with deterministic logic.

The design insight: incremental itemized updates instead of LLM-rewriting one big prompt — rewriting causes context collapse (good content silently lost) and brevity bias. Treat context like a database, not a blob.

Rung: optimizer code

MCE & Meta-Harness — optimize the mechanism, not the artifact

meta context engineering · ye 2026

inner: c*ₛ = argmax_c J_train(c; s) best context, given skill s outer: s* = argmax_s J_val(c*ₛ) best skill, on held-out val skill s = ρₛ static: prompts, kb, libs + Fₛ dynamic: search/filter/format

Skills stored as files (skill.md + code); a meta-agent evolves them by crossover over the skill database.

meta-harness · lee 2026

  • A proposer coding agent writes new harnesses as code — a harness for optimizing harnesses.
  • Full execution history lives on the file system; agents grep/cat it rather than stuffing one prompt.
  • Each candidate = {source, scores, trajectories}; keep the Pareto frontier.
  • Improved on TerminalBench-2 even starting from strong baselines (Terminus-KIRA, Terminus-2).

MCE separates the mechanism (how context is managed) from the artifact (the context items). Bi-level optimization: inner loop learns the best context for a given skill; outer loop evolves the skills themselves against validation performance. Skills = static components + dynamic operators, instantiated as file collections, edited with the standard tool set {Read, Write, Edit, Bash, Glob, Grep, TodoWrite} — literally the Claude Code toolkit.

Meta-Harness closes the loop: once harness design is an executable search space, a strong coding agent can explore it the way human engineers do. Output is a Pareto frontier of harnesses, not a single winner.

Rung: workflow

Expert-designed workflows: automating research itself

ScientistOne
Verifiability as the design constraint: every claim — citation, number, method, conclusion — must trace to evidence via Chain-of-Evidence checks.
Autodata
Agent manages a challenger, a weak solver, a strong solver, and a verifier to synthesize training data at "just right" difficulty: strong succeeds, weak fails.

Caveat: Autodata's synthesized tasks improve the weak solver — indirect distillation, not yet full self-improvement.

These are handcrafted-by-experts workflows — the rung below self-modifying harnesses. AI Scientist runs the whole pipeline; it landed in Nature 2026, but note the later slide: producing papers is not the same as producing discoveries.

Autodata's difficulty targeting is a lovely trick: a task is "just right" for training when the strong solver passes and the weak one fails. The challenger's prompt is iteratively updated from solver+verifier feedback. But the gains flow to the weak solver — that's distillation, not RSI.

Rung: workflow — searched, not handcrafted

ADAS & AFlow: agent design as a search problem

ADAS · hu, lu & clune 2025

  • A meta-agent programs new agents in code, drawing inspiration from an archive of prior designs.
  • Seed archive with simple agents (CoT, self-refine); describe → implement → two self-refine novelty checks.
  • Evaluate; successful designs join the archive; repeat.

AFlow · zhang 2025

  • Workflow = graph: nodes are LLM calls, edges are code logic.
  • Optimized with MCTS: select a node balancing score vs exploration, expand with an LLM-modified workflow, execute, evaluate, keep if better.
  • Beats manual designs and ADAS on QA, code, and math benchmarks.

ADAS formulates agent design itself as the optimization problem — "meta agent search". The archive plays the role of an evolutionary population; novelty checks fight duplication.

AFlow makes the search space crisp: workflows as graphs, searched with Monte Carlo Tree Search under an N-round budget. Selection balances exploitation of good workflows with exploration.

Bridge to next slides: workflows are still *structures around* a fixed harness. The next move is letting the system edit the harness code itself.

Rung: harness code

Self-Harness: propose → evaluate → accept, with a regression gate

python3 demo/run.py 4 — watch a reward-hack proposal get caught by the held-out gate

Three stages: (1) weakness mining — cluster failures into verifier-grounded patterns, root causes not surface errors; (2) bounded proposals — the same model, given editable harness surfaces + failure patterns + past attempts, prefers narrow recurrent fixes; (3) validation on held-in AND held-out, accept only with no regression.

Weakness mining, concretely (this is what's running live in demo 4): every held-in failure gets bucketed by a cluster label — whitespace/decimals, currency-symbol, thousands-separator — instead of being treated as its own one-off bug. The loop then targets whichever cluster has the most failures, so one bounded fix knocks out several failures at once instead of whack-a-mole patching inputs one at a time. That's the "root cause, not surface error" distinction: the error message says "expected 3.10 USD, got $3.1"; the cluster label says the actual mechanism to fix is currency-symbol parsing.

STOP (Zelikman 2023) is the ancestor: an improver program that improves itself, judged by meta-utility averaged over tasks. It discovered genetic algorithms, beam search, simulated annealing on its own.

DEMO 4 runs this loop for real: a candidate rule that hardcodes "USD" aces the held-in suite (6/6!) but regresses held-out non-USD cases — rejected. The gate, not good intentions, catches the hack.

Rung: harness code — population-scale

Evolutionary search: survival of the fittest diff

wire the loop: pool ← candidate programs repeat: parent ← sample(pool) # rank ∝ fitness, # penalize offspring count child ← LLM_diff(parent) # mutation score ← evaluate(child) # run it for real pool ← keep_fittest(pool + child) # EVOLVE-BLOCK-START …only this region is mutable… # EVOLVE-BLOCK-END
AlphaEvolve
Frozen LLM proposes diffs to a program pool; meta-prompts co-evolve. New math & algorithm discoveries.
ShinkaEvolve
Sample-efficiency: smart parent sampling, embedding-based novelty rejection, a meta-scratchpad of winning patterns.
Darwin Gödel Machine
The agent evolves its own harness repo. SWE-bench Verified 20% → 50%, Polyglot 14.2% → 30.7%.

Works when fitness is fast & objective (kernels, contests, schedulers). Struggles when evaluation is slow, ambiguous, or a vibe.

python3 demo/run.py 5 — evolve real code in an EVOLVE-BLOCK until MSE = 0

Evolutionary search fits harness optimization because the space is huge and weirdly shaped, gradients don't exist, but evaluation is cheap. Lineage: Promptbreeder and GEPA did this for prompts; AlphaEvolve for programs; DGM for the agent's own harness code.

DGM's numbers are the headline: agents that rewrote their own scaffolding went from 20% to 50% on SWE-bench Verified — matching or beating hand-crafted agents.

DEMO 5: symbolic regression — mutate the EVOLVE-BLOCK, exec the program, score MSE against examples, select. Converges to the exact hidden formula around generation 30. Emphasize: seeded, deterministic, every candidate genuinely executed.

The caveat that keeps everyone honest

Recursion needs a strong base model

  • STOP's key finding: the self-improving improver worked with GPT-4 — and degraded with GPT-3.5 and Mixtral.
  • Harness improvement multiplies capability; it cannot substitute for it.
  • Long term, many harness tricks may internalize into model behavior — as prompt engineering did after instruction tuning — but tool interfaces and external context will remain.
capability = model^harness weak model, great harness: loops forever, confidently strong model, great harness: compounding improvement

This is the slide to slow down on. STOP improved recursively only above a capability threshold; below it, self-modification made things worse. So "just add a better harness" is not a free lunch — the two layers gate each other.

Weng's prediction: the harness layer and core intelligence co-evolve. Some patterns (like context compaction) may get absorbed into models, the way instruction tuning absorbed most prompt hacks — but you'll always need to specify goals and expose tools.

Reality check — autonomous research today

Six recurring failure modes (Trehan & Chopra 2026: 4 ideas in, 1 paper out)

  • Training-data defaults — old libraries, stale commands, assumptions never checked against the actual repo.
  • Implementation drift — under pressure, the model quietly swaps the proposed method for a familiar simpler one.
  • Context degradation — long-horizon detail lost without persistent artifact logging.
  • Over-optimism — “p-hacking and eureka-ing”: numerical duct tape, victory declared amid noise.
  • Missing domain intelligence — no tacit craft: complexity estimates, plausibility checks, relevant baselines.
  • Weak scientific taste — experiments run, but the right questions never get asked.

Producing a plausible manuscript ≠ producing a discovery. Fabricated citations, drifted implementations, and weak results all still compile.

Trehan & Chopra gave LLMs minimal scaffolding (read_file, write_file, llm_search, list_files) and asked them to go idea → paper. One of four ideas made it. These six patterns explain the gap.

The "p-hacking and eureka-ing" phrase is from Bubeck et al. 2025 (GPT-5 science-acceleration experiments). Engineers will recognize every one of these from code agents too — same pathologies, different domain.

Note how failure modes 1–4 are harness-addressable (grounding, journaling, verification), while 5–6 are capability gaps. That maps back to the caveat slide.

Open problems

Seven bottlenecks between here and full RSI

  1. Weak, fuzzy evaluators — most real tasks lack a fast, precise verifier; loops need one to function.
  2. Context & memory lifecycle — memory should become core intelligence, not a bolted-on subsystem.
  3. Negative results — literature (and training data) hides failures; agents must learn to abandon hypotheses.
  4. Diversity collapse — evolution and RL exploit known winners; open-ended research needs paths that look worse first.
  5. Reward hacking — evaluators & permissions must live outside the loop: held-out tests, trace audits, human review.
  6. Long-term goals — repo health, migrations, ownership: rarely capturable in sandboxed RL rewards.
  7. The role of humans — move up the stack, not out of the loop: oversight at the right abstraction and timing.

These are Weng's closing seven. Themes to draw out for an engineering audience:

#1 and #5 are two sides of one coin — the loop is only as good as its judge, and the judge must be un-gameable and un-editable by the thing being judged (demo 4 showed a miniature of this).

#4: mechanisms like novelty rejection (ShinkaEvolve) and archive sampling (ADAS/DGM) exist precisely to fight collapse.

#6 hits home: coding agents boost daily productivity, but "keep a 1000-engineer repo healthy for five years" is not yet an optimizable reward.

#7 is the article's closing note: technology should serve humanity's better future, not vice versa.

Takeaways for working engineers

What to steal for your own systems

  • Build the loop first. Plan → act → observe → improve, with verification inside the loop — the loop enforces honesty, not the model.
  • Put state on disk. Context is working memory; anything worth keeping goes to files the next session can read.
  • Fan out through files. Sub-agents that report via artifacts stay inspectable; transcripts don't.
  • Gate every self-modification. Held-out suites, bounded diffs, regression-or-reject — demo 4 in one line.
  • Keep evaluators outside the blast radius. Whatever the loop optimizes, the judge must not be editable by it.
  • Expect the ladder to keep climbing. Today you tune prompts; next quarter, the thing tuning the prompts.
